# 场景：业余科幻小说翻译，逐章英译中

> **用户描述**：我是科幻小说爱好者，业余做英文科幻小说的中文翻译，翻没引进中国的单行本，发博客和论坛。流程是：拿到英文 PDF/EPUB → 逐章阅读理解 → 翻译成中文 → 校对（逐句对比中英文，查漏译误译）→ 润色 → 排版发布。一本 300 页翻 2-3 个月，每天 2-3 小时。最痛苦的不是翻译本身，而是校对——翻完一章后要逐句对比中英文，检查有没有漏掉的句子、翻错的术语、前后不一致的人名地名。极其枯燥但不能省。

---

## 你现在的问题

> 你的翻译能力没有被卡住，你的校对耐心正在被耗尽。翻译是创造性的，你享受那个过程；但校对是拿着显微镜在两份文本之间来回扫描——本质上是人肉 diff 工具。你用大脑最昂贵的注意力资源，去做一件机器可以碾压你的事：逐句对齐、逐词比对、跨章追踪术语一致性。更隐蔽的问题是：因为校对太痛苦，你可能在不自觉地降低校对标准——"这段应该没问题吧"——而漏译和术语不一致恰恰是读者最先发现的质量问题。

---

## 闭环拆解

| 步骤 | 现在谁在做 | 具体动作 | 状态 |
|------|-----------|---------|------|
| **调研** | 人肉 | 拿到 PDF/EPUB → 想办法提取文本 → 逐章阅读理解原文 | 提取文本是纯苦力 |
| **决策** | 人（有判断价值） | 理解原文语义、选择译法、处理双关语和文化隐喻 | 核心创造性工作，不可替代 |
| **规划** | 人肉 | 脑子里记着人名地名术语怎么翻的，靠记忆保持一致 | 没有显性术语表，全靠脑记 |
| **执行** | 人（有判断价值） | 写出中文译文 | 核心创造性工作 |
| **反馈** | 人肉 | 翻完一章后逐句对比中英文，找漏译、误译、术语不一致 | **意愿瓶颈 + 吞吐量瓶颈** |
| **复盘** | 缺失 | 没有系统性回顾哪类句子容易漏、哪些术语经常翻错 | 同类错误反复犯 |

六步里，决策和执行是你真正的价值所在。但反馈环节（校对）占了你 40% 的时间，并且是最消耗意愿的一步。规划环节（术语一致性）没有工具支撑，全靠记忆，随着翻译推进越来越容易出错。复盘环节完全缺失。

---

## Claude Code 怎么驱动这一切

**机制**：Claude Code 运行在你的电脑上，读取你项目目录里的 CLAUDE.md 指令文件，知道这本书的术语表、你的翻译偏好、每章的文件结构。它用 `pandoc` 把 EPUB 转成按章分割的 Markdown 文本，用 Python 脚本做中英文逐句对齐和比对，用自身的语言理解能力做语义级别的漏译和误译检测。它不是一个翻译机器——你是译者，它是你的校对搭档。

**故事**：

你刚翻完第七章，3200 字，花了两个小时，脑子已经有点糊了。按以前的流程，你得再花一个小时逐句对着原文检查——但现在你打开终端，跟 Claude Code 说一句："校对第七章"。

它先读取 `chapters/ch07-en.md`（原文）和 `chapters/ch07-zh.md`（你的译文），把两份文本按段落对齐。然后它逐段扫描，做三件事：

**第一，查漏译。** 它把原文每一段和译文每一段对齐后，检查是否有原文段落在译文中完全没有对应内容。它发现第 7 段有一句 "The station's emergency protocols hadn't been updated since the Kepler incident" 在你的译文里没有出现——你翻到那段时可能被上下文带走了，漏掉了这句。它列出来：

> 漏译疑似 1 处：
> 原文 P7: "The station's emergency protocols hadn't been updated since the Kepler incident."
> 建议补译位置：译文第 7 段末尾

**第二，查术语一致性。** 它读取 `glossary.jsonl`——你的术语表，记录着这本书里所有专有名词的译法。它扫描这一章，发现你在第 3 段把 "quantum entanglement drive" 翻成了"量子纠缠引擎"，但术语表里记录的是"量子纠缠驱动器"，前六章一直用的也是"驱动器"。它标出来：

> 术语不一致 1 处：
> "quantum entanglement drive" → 本章译为"量子纠缠引擎"，术语表及前文均为"量子纠缠驱动器"
> 建议：统一为"量子纠缠驱动器"

**第三，查可疑误译。** 它逐句理解原文和译文的语义，发现第 12 段有一句 "She couldn't afford to lose another cycle" 你翻成了"她不能再失去一个循环"——但结合上下文，这里的 "cycle" 指的是空间站的轮班周期，不是抽象的"循环"。它提出来让你判断：

> 可疑译法 1 处：
> 原文: "She couldn't afford to lose another cycle."
> 当前译文: "她不能再失去一个循环。"
> 上下文暗示 cycle 指轮班周期，建议考虑："她不能再浪费一个轮班周期了。"
> （请确认，你对上下文的理解可能比我更准确）

三分钟，它把结果整理好。你扫一眼——漏译那句确实漏了，补上；术语那个改回"驱动器"；"cycle"的翻译你想了想，觉得它说得对，改了。全程 5 分钟，替代了原来一小时的逐句人肉扫描。

然后它问你："这章出现了两个新术语：'phase-locked orbit'（你译为'锁相轨道'）和 'biosynthetic membrane'（你译为'生物合成膜'），要加入术语表吗？"你说"加"，它自动追加到 `glossary.jsonl`，下一章开始就会检查这两个词的一致性。

---

## 你的工作会变成什么样

你每天晚上翻译一章，翻完后说一句"校对这章"。三分钟后你拿到一份清晰的校对报告：几处漏译、几处术语不一致、几处可疑译法。你花 5-10 分钟过一遍，该补的补，该改的改，拿不准的留着下次通读时再看。

校对从"一小时逐句死磕"变成了"五分钟扫报告拍板"。

更重要的是，术语表是活的。你从第一章开始积累，到第十章的时候，"Kepler Station"是"开普勒空间站"还是"开普勒站"，"the Drift"是"漂流区"还是"迷失带"——这些决定都被记录下来了，不用靠脑子记。整本书译完，你手里多了一份完整的术语表，以后翻这个作者的下一本书可以直接复用。

2-3 小时的日常工作变成：翻译 1.5-2 小时（这部分没变，也不该变）+ 校对审核 10 分钟。省出来的 50 分钟，你可以拿去润色译文——让中文读起来更自然，这才是你作为译者真正想花时间的地方。

一本 300 页的书，从 2-3 个月缩短到 1.5 个月左右。不是因为翻得更快了，而是因为校对不再拖后腿了。

## 怎么搭建

### 1. 准备工具

```bash
# pandoc（文档格式转换，把 EPUB 转成按章 Markdown）
# 你已经装了 pandoc 3.9，不需要额外安装
pandoc --version

# Calibre（处理 PDF 转换，提供 ebook-convert CLI）
brew install --cask calibre
# 安装后 ebook-convert 命令在 /Applications/calibre.app/Contents/MacOS/ebook-convert

# poppler（PDF 文本提取备选方案）
brew install poppler
# 安装后可用 pdftotext 命令

# Python 依赖（文本处理）
pip install regex
```

### 2. 创建项目目录

每本书一个项目：

```bash
mkdir ~/translation/book-name && cd ~/translation/book-name
mkdir -p source chapters output
```

目录结构：

```
~/translation/book-name/
├── CLAUDE.md              # Agent 指令
├── glossary.jsonl          # 术语表（逐行 JSON）
├── source/
│   └── book.epub           # 原版电子书
├── chapters/
│   ├── ch01-en.md          # 第一章原文（pandoc 提取）
│   ├── ch01-zh.md          # 第一章译文（你写的）
│   ├── ch02-en.md
│   ├── ch02-zh.md
│   └── ...
└── output/
    └── review-ch01.md      # 校对报告
```

### 3. 第一次使用：提取原文

```bash
cd ~/translation/book-name
claude
# 说："把 source/book.epub 按章节拆分成 Markdown，放到 chapters/ 目录下"
# Claude Code 会用 pandoc 处理 EPUB，按章节标题拆分成 ch01-en.md、ch02-en.md...
# 如果是 PDF，它会用 pdftotext 或 ebook-convert 先转格式再拆分
```

### 4. 编写 CLAUDE.md

```markdown
# 科幻小说翻译校对 Agent

## 身份
你是英译中科幻小说的校对助手。你不翻译，你只检查译者已完成的译文。
译者对原文的理解和译法选择拥有最终决定权，你的角色是发现问题、提出建议，不是代替判断。

## 项目结构
- source/：原版电子书文件
- chapters/：按章分割的原文（ch{NN}-en.md）和译文（ch{NN}-zh.md）
- glossary.jsonl：术语表
- output/：校对报告

## 术语表格式（glossary.jsonl）
每行一个 JSON 对象：
{"en": "quantum entanglement drive", "zh": "量子纠缠驱动器", "note": "小说核心设定，全书统一", "first_appear": "ch01"}

## 校对流程

当译者说"校对第 N 章"时：

### Step 1：对齐
读取 ch{N}-en.md 和 ch{N}-zh.md，按段落对齐原文和译文。

### Step 2：漏译检查
逐段比对，找出原文中有但译文中缺失的句子或段落。
输出格式：
- 原文位置（第几段第几句）
- 原文内容
- 建议补译位置

### Step 3：术语一致性检查
读取 glossary.jsonl，扫描本章译文中所有术语表收录的词汇：
- 是否与术语表一致
- 是否与前文章节一致
- 如果不一致，列出差异和建议

### Step 4：可疑误译检查
逐句理解原文和译文的语义对应关系，标记：
- 明显的语义偏差（翻反了、理解错了）
- 需要结合上下文判断的多义词
- 科幻术语的翻译是否符合上下文设定
对每一处可疑点，给出原文、当前译文、你的理解和建议，但明确标注"请译者确认"。

### Step 5：新术语识别
扫描本章中出现的新专有名词（人名、地名、科技术语、虚构概念），如果不在术语表中：
- 列出英文原文和译者本章使用的译法
- 询问译者是否加入术语表

### Step 6：输出报告
将结果写入 output/review-ch{N}.md，格式清晰，按严重程度排序：
1. 漏译（最严重）
2. 术语不一致
3. 可疑误译
4. 新术语待确认

## 其他能力

### 原文提取
当译者说"提取原文"时：
- EPUB：用 pandoc 转成 Markdown，按章节标题（# 或 ## ）拆分
- PDF：先用 pdftotext 提取纯文本，再按章节标记拆分
- 输出到 chapters/ch{NN}-en.md

### 术语表管理
- "查术语 XXX"：在 glossary.jsonl 中搜索
- "改术语 XXX → YYY"：更新术语表并提醒全书替换
- "导出术语表"：输出为可读的 Markdown 表格

### 全书一致性检查
当译者说"全书检查"时：
- 扫描所有已完成章节的译文
- 检查术语表中每个词在各章是否一致
- 检查人名地名是否有变体写法
- 输出全书一致性报告

### 润色辅助
当译者说"润色第 N 章"时：
- 只关注中文表达的自然度，不改变译者的翻译选择
- 标记读起来别扭的句子（欧化语法、过长从句、不自然的被动句）
- 给出润色建议，但保留译者的风格和用词偏好
```

### 5. 初始化术语表

```bash
# 创建空术语表
echo '' > glossary.jsonl
```

第一章翻完后，Claude Code 会帮你识别出现的专有名词，你确认译法后自动写入。后续章节自动检查一致性。

### 6. 启动验证

```bash
cd ~/translation/book-name
claude
# 第一步："把 source/book.epub 提取原文，按章节拆分到 chapters/"
# 确认拆分结果是否正确（章节边界对不对、文本有没有乱码）

# 翻完第一章后，把译文存为 chapters/ch01-zh.md
# 然后说："校对第 1 章"
# 第一次全程盯着，确认：
#   ① 段落对齐是否准确
#   ② 漏译检测是否靠谱（可以故意漏一句测试）
#   ③ 术语识别是否合理
```

## 运行模式：交互模式为主

| 模式 | 触发方式 | 覆盖场景 | 你的介入 |
|------|---------|---------|---------|
| **交互模式** | 你主动启动 | 逐章校对、术语讨论、润色辅助 | 你坐在电脑前，审核每一条建议 |
| **定时模式** | 全书译完后一次性触发 | 全书术语一致性检查 | 看报告，批量修正 |

翻译是深度创造性工作，不适合无人值守。Claude Code 在这个场景里是你的"第二双眼睛"，而不是自动流水线。你翻完一章，它帮你检查一章，你审核确认——这个节奏本身就是自然的。

## 元层：它会越来越好

**术语表是活的知识库**：每一章的校对过程都在丰富术语表。到全书中段，术语表已经足够完善，校对报告里的"新术语"会越来越少，"术语不一致"也会越来越少，因为你已经把所有决策显性化了。这份术语表跟着你——下一本同系列的书、同一个作者的书，直接复用。

**漏译模式学习**：跑了几章之后，你会发现自己的漏译有规律——比如你特别容易漏掉括号里的补充说明、或者对话中间的动作描写（"said"后面跟的那小半句）。Claude Code 可以在校对报告的末尾加一段统计："本章漏译 2 处，均为对话间插入的动作描写，与第 3、5 章的漏译模式一致。" 你知道了自己的弱点，翻下一章时会特别留意这类句子。

**校对规则迭代**：如果你发现 Claude Code 总是误报某类"可疑误译"（比如你有意为之的意译），告诉它"这类不用报"，它更新 CLAUDE.md 里的规则，下次不再打扰。误报率会从第一章的 30% 逐步降到 5% 以内。

**风格偏好沉淀**：润色环节中，你接受的建议和拒绝的建议，本身就在训练它理解你的翻译风格——你喜欢短句还是长句、你倾向于保留原文结构还是大幅重组、你对科幻术语是音译还是意译。这些偏好可以写进 CLAUDE.md 的"译者风格"部分，让后续建议越来越贴合你的习惯。
